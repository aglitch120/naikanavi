## 2. 記事作成ルール

### 2.1 タイトル（titleタグ）
- **40文字以内**
- **主要キーワードを前方に配置**
- 【】を使う場合は冒頭に
- 年度を入れる場合は「2026年最新」「2026年版」

```
✅ 良い例
【2026年最新】J-OSLER病歴要約の書き方完全ガイド｜Accept率を上げるコツ

❌ 悪い例
病歴要約の書き方について解説します｜J-OSLER対策
```

### 2.2 メタディスクリプション
- **120文字以内**
- 「誰の」「何の悩みを」「どう解決するか」を含める
- CTAを含める（〜を解説、〜がわかる）

```
✅ 良い例
J-OSLER病歴要約の書き方を査読委員経験者が徹底解説。総合考察・全人的視点の書き方から、差し戻しを防ぐコツまで。テンプレート付きで今日から使えます。
```

### 2.3 見出し構成
```
H1: タイトル（1つのみ）
├── H2: 大見出し（5〜8個）
│   ├── H3: 中見出し
│   │   └── H4: 小見出し（必要な場合のみ）
```

- H2に関連キーワード・共起語を含める
- H2の最初に「結論」「まとめ」を持ってこない（最後に配置）

### 2.4 本文の書き方

#### 冒頭（リード文）
- **100字以内で「誰の何の悩みを解決するか」を明示**
- 読者の共感を得るフック
- 記事を読むメリット

```
✅ 良い例
「J-OSLERの病歴要約、何度書いても差し戻される...」そんな悩みを抱えていませんか？
この記事では、査読委員を8年務めた私が、Acceptされる病歴要約の書き方を徹底解説します。
```

#### 本文構成
- 一文は60文字以内を目安
- 3〜4文ごとに改行
- 箇条書きは5個以内
- 具体例・データを必ず入れる

#### 文字数目安
| 記事タイプ | 文字数 |
|-----------|--------|
| 通常記事 | 3,000〜5,000字 |
| ピラー記事 | 5,000〜8,000字 |
| ツール紹介 | 2,000〜3,000字 |
| Q&A形式 | 1,500〜2,500字 |

### 2.5 画像・図解
- アイキャッチ画像必須（1200×630px推奨）
- 本文中に2〜3枚の図解
- ALT属性に必ずキーワードを含める
- ファイル名は英数字のみ（josler-byoreki-template.png）

### 2.6 内部リンク
- **1記事あたり3〜5本**
- 関連記事への自然な導線
- ピラー記事とクラスター記事を相互リンク
- アンカーテキストにキーワードを含める

### 2.7 CTA配置
| 位置 | タイプ | 目的 |
|------|-------|------|
| 記事上部 | テキストリンク | 即座にツールへ |
| 記事中盤 | バナー | 読み進めた人への提案 |
| 記事下部 | 大型CTA | 記事完読者への訴求 |

## 11. 記事作成の品質チェック（2026年3月追加）

### 11.1 文字数の最適化
- **ピラー記事**: 5,000〜8,000字（本文のみ、frontmatter除く）
- **通常記事**: 3,000〜5,000字
- **ツール紹介**: 2,000〜3,000字
- 長ければ良いわけではない。**冗長な繰り返しは削除**し、検索意図を満たす最小限の情報量を目指す
- 同じ情報を別の言い回しで繰り返さない

### 11.2 リンク戦略（必須）

#### 内部リンク（3〜5本/記事）
- 関連する既存記事へ**自然な文脈で**リンク
- アンカーテキストにキーワードを含める
- ピラー↔クラスター間の相互リンクを意識

#### 外部リンク（2〜3本/記事）
- **公式ソースを必ずリンク**（日本内科学会、J-OSLER公式ページ等）
- 権威性の高い一次ソースを優先
- `target="_blank" rel="noopener noreferrer"` をつける
- 競合サイトへは原則リンクしない

### 11.3 CTA設計（3箇所必須）

#### 冒頭CTA（inline）
- **ユーザーが「え、マジ？」と思う訴求**にする
- 「テンプレートを無料ダウンロード」ではなく「AIが病歴要約の下書きを30秒で生成」のような具体的ベネフィット
- 数字を入れる（30秒、10分の1、など）

#### 記事中インラインCTA（2〜3箇所）
- 自然な文脈でハイパーリンクとして設置
- 引用ブロック（`>`）で目立たせる箇所は1箇所まで
- 残りは文中に自然にリンクを溶け込ませる

#### 末尾CTA（large/ヒーロー風）
- SVGイラスト付き
- バッジ（ブランド名）
- 3つの特徴・メリットをチェックマーク形式で
- 明確なボタン（「無料で試してみる」等）
- 補足テキスト（「無料トライアルあり。クレジットカード不要。」等）

### 11.4 画像・図解ルール

#### SVG図解（2〜3点/記事）
- DESIGN_SYSTEM.mdのカラーパレットに準拠
- カード内にミニアイコン（クリップボード、ペン、チェックマーク等）でポップさを出す
- `© iwor iwor.jp` のフッターを入れる（`©` は文字リテラルで記述）
- ALT属性に具体的な説明を入れる

**SVG作成時の必須ルール（XMLパースエラー防止）：**
- `&copy;` `&trade;` `&reg;` 等のHTMLエンティティは使用禁止 → `©` `™` `®` の文字リテラルを使う
- SVGはXMLなので `&amp;` `&lt;` `&gt;` `&quot;` `&apos;` の5つ以外のエンティティは未定義エラーになる
- `rx` `ry` 属性は単一の数値のみ（`rx="6"` ○ / `rx="0 0 6 6"` ✕）
- 作成後は必ずXMLバリデーションを実行: `python3 -c "import xml.etree.ElementTree as ET; ET.parse('file.svg')"`

#### OGP画像
- 新記事追加時は `app/blog/[slug]/opengraph-image.tsx` のtitleMapに追記すること

### 11.5 目次の長さへの配慮
- H2は5〜7個を目安（目次が長くなりすぎない）
- H3は各H2配下に2〜3個まで
- 目次コンポーネントは折りたたみ式（アクティブH2のH3のみ表示）で実装済み

### 11.6 OGP画像（slugToMetaマップ）の更新
- 記事を新規追加したら、必ず `app/blog/[slug]/opengraph-image.tsx` の `slugToMeta` にslug・title・subtitle・categoryを追加する
- OGP画像デザイン: **緑背景（#1B4F3A）**、カテゴリバッジ（白半透明）、大きな白文字タイトル、グリーンサブタイトル、右に装飾円
- Noto Sans JP Boldフォントを動的fetchして太字を正しくレンダリング

### 11.7 アイキャッチ画像
- OGP画像を記事ページ冒頭にアイキャッチとして自動表示
- **配置順序: パンくず → カテゴリバッジ+◯分で読める → アイキャッチ → タイトル+メタ情報 → CTA → 目次 → 本文**
- `rounded-xl overflow-hidden` でデザイン統一

### 11.8 冒頭目次（InlineTableOfContents）
- **モバイルのみ**（`lg:hidden`）で記事冒頭に折りたたみ式目次を表示
- デフォルトではH2を4件までプレビュー表示、「...他N件を表示」で展開
- 展開時はH2+H3すべてを表示、クリックで該当箇所にスクロール後自動で閉じる
- **PCはサイドバー目次（スクロール追従）のみ**

### 11.9 シェアボタンのモバイル対応
- `flex-wrap`で折り返し可能に
- タップターゲット最小44px（`min-h-[44px]` + `px-3 py-2`）
- 「共有:」ラベルはモバイルでは単独行

### 11.10 フッター
- main + footerのコンテナは `px-6 sm:px-8` で余白統一
- フッター下部（ロゴ・リンク・コピーライト）は**左揃え統一**（`items-start`、`text-center`不使用）

### 11.11 CTAのアニメーション
- 末尾CTA（large）: `ctaFloat` アニメーション（0.6s, translateY 12px→0, ease-out）
- 冒頭CTA（inline）: `ctaFloat` アニメーション（0.4s, 同上）
- サブリミナルなレベルの微細な浮き上がり。派手にしない

### 11.12 スマホ冒頭CTAの簡潔さ
- モバイルではdescription（説明文）を非表示（`hidden sm:block`）
- タイトル + ボタン のみのシンプルな構成
- パディング・フォントサイズもPC比で縮小

### 11.13 関連記事カード（ArticleCard compact）
- OGP画像をサムネイルとして表示（`/blog/[slug]/opengraph-image`）
- カード構成: サムネ画像 → カテゴリバッジ → タイトル

### 11.14 マークダウン太字の注意（重要）
remarkパーサーの仕様で、以下のパターンは**太字として認識されない**：
- `**「テキスト」**` → NG（`**`の内側に`「」`があると壊れる）
- `**テキスト（説明）**の` → NG（閉じ`**`の直後に日本語文字が空白なしで続く）
- `**テキスト）**と` → NG（閉じ`）`の直後に`**`、さらに日本語が続く）

**正しい書き方**：
- `「**テキスト**」` → OK（括弧を`**`の外に出す）
- `**テキスト（説明）** の` → OK（閉じ`**`の後にスペース）
- `**テキスト** と **他のテキスト** が` → OK（前後にスペース）
- `**テキスト**は` → OK（内側に括弧がなければ問題なし）

公開前にプレビューで太字が正しくレンダリングされているか必ず確認すること。

### 11.15 タグ
- `/blog/tag/[tag]` でタグページ実装済み（2026-03-10）
- 記事ページのタグは `<Link>` でタグページへ遷移
- ホバー時に `bg-acl text-ac` でインタラクション表示
- `generateStaticParams` で全タグをSSG対応

### 11.16 Edge Runtimeファイルの日本語制限
- `opengraph-image.tsx` 等 `export const runtime = 'edge'` を含むファイルでは、**オブジェクトキーにASCII文字のみ使用**（esbuildが中黒 `・` 等のUnicodeキーをパースできないため）
- 値（value）に日本語を使うのはOK
- カテゴリ等の日本語表示が必要な場合は、ASCIIスラッグ → 日本語のマッピング関数を使う

### 11.22 テンプレート・例文の医学的記載ルール

記事内のテンプレートや使用例（概略・自己省察の例文）は、**医学論文・症例報告の記載作法** に従うこと。読者がそのままJ-OSLERにコピペして使うことを想定しているため、ブログ本文とは異なるフォーマットが必要。

#### 表示形式（重要）
- テンプレート・使用例（概略の例・自己省察の例）は **コードブロック（` ``` `）** で統一する（黒背景で本文と明確に区別）
- コードブロックは自動的に **コピーボタン付き・レスポンシブ折り返し表示** で描画される（`CopyableCodeBlock` コンポーネント）
- ❌ 引用ブロック（`>`）をテンプレートや例文に使わない（CTAとの区別がつかなくなる）
- ❌ プレーンテキストでテンプレートや例文を書かない（本文との区別がつかなくなる）
- ❌ 横スクロールが必要になるような `overflow-x` 設定は使わない
- 引用ブロック（`>`）は **CTA・注意書きのみ** に使用する
- テンプレートと使用例で表示形式が混在しないよう、必ず全箇所をコードブロックに揃えること

#### 句読点
- テンプレート・例文内では「、」「。」ではなく **「,」「.」** を使用する
- ブログ本文（解説パート）は通常どおり「、」「。」でよい

#### 動詞の選び方
- ❌ 「〜を認めた」（主語が曖昧・受動的）
- ⭕ 「〜であった」「〜を呈した」「〜を示した」「〜を指摘した」（より客観的・医学的）
- 例: 「CRP高値を認め」→「CRP高値であり」「CRP高値を示し」

- 薬剤名は一般名を使用（商品名は避ける）

### 11.29 記事の「深み」と権威性（2026年3月追加・最重要）

#### 品質基準：3月10日以降の記事レベルを常に維持すること

iworの記事は「医師が求める具体的な答えを提供する臨床・キャリア支援メディア」である。**一般的な情報をまとめただけの記事は公開しない。** 以下の基準を全て満たすこと。

#### ① 具体性・専門性（最重要）

- **抽象的な説明を書かない**。「バーンアウトには注意が必要です」ではなく「専攻医3年目の2〜3月、病歴要約の残り10篇と当直月6回が重なる時期が最もバーンアウトリスクが高い」レベルの具体性
- **数値・制度・固有名詞を入れる**: 「何回」「何件」「何月」「具体的な要件名」
- **読者の実際の体験に即したシナリオ**を書く（「〜することで〜が解決した」という具体的な文脈）
- 公式ソースや根拠がある場合は必ず引用。「〜とされています」は使わない

#### ② 太字によるメリハリ（必須）

散文パートには **1段落あたり1〜2箇所** の太字を入れること。太字は「重要な結論」「重要な用語」「読者が最も知りたいこと」に絞る。

**✅ 良い太字の使い方：**
- `**7期生以降は160→120症例に変更**された。` → 制度変更の重要情報
- `差し戻しが発生すると**数ヶ月の遅延**につながる。` → 読者が最も注意すべき結論
- `**週1〜2時間のJ-OSLER作業時間**を確保することが、3年間を通じた最善策。` → 具体的な行動指針

**❌ 悪い太字の使い方：**
- 箇条書きの各行の冒頭を全て太字にする（ただの装飾で意味がない）
- 1段落に太字が5箇所以上ある（何が重要かわからなくなる）

#### ③ 文字の壁を作らない

- **段落は4文以内**を目安。5文以上になったら分割する
- 長い段落の後には、短い1〜2文の段落や引用ブロックを置いてリズムを変える
- 「〜です。〜ます。〜です。〜ます。」の単調なリズムを避ける。疑問文・短文・箇条書きを適切に混ぜる

#### ④ SVGの品質基準

3月10日レベルのSVGを維持すること：

```xml
<defs>
  <filter id="fc1" x="-4%" y="-4%" width="108%" height="108%">
    <feDropShadow dx="0" dy="2" stdDeviation="4" flood-color="#000" flood-opacity="0.08"/>
  </filter>
  <marker id="arrow" viewBox="0 0 10 10" refX="5" refY="5"
    markerWidth="6" markerHeight="6" orient="auto-start-reverse">
    <path d="M 0 0 L 10 5 L 0 10 z" fill="#1B4F3A"/>
  </marker>
</defs>
```

- カードにはドロップシャドウ (`filter="url(#fc1)"`) を使用
- フロー図にはアローマーカー (`marker-end="url(#arrow)"`) を使用
- 背景は `#F5F4F0`（ウォームグレー）または `#F5F7F5`（クールグレー）
- テキストの `font-family` は `sans-serif`、重要箇所は `font-weight="bold"`
- **MDX内のSVGにはHTMLコメント `<!-- -->` を書かない**（MDXパーサーエラーになる）

#### ⑤ 権威性の構築

- 読者が「この記事を書いた人は本当に詳しい」と感じるレベルの情報密度
- 「こういう失敗をしやすい」という**具体的なよくある間違い**を記述する
- 「実際にやってみると〜」「〜の場面では〜」という**現場感のある表現**を使う
- 公式ページへの外部リンクを入れる（[日本内科学会](https://www.naika.or.jp/nintei/j-osler/)等）

### 11.30 ファクトチェック義務化ルール（2026年3月追加・最重要）

記事内の制度・数字・資格要件は**必ずWeb検索で最新情報を確認**してから記載すること。AIの学習データは古い可能性が高く、制度改正を反映していない場合がある。以下のカテゴリは特に要注意。

#### ① 絶対にWeb検索で確認すべき項目（公開前の確認必須）

| カテゴリ | 例 | 確認先 |
|---------|---|--------|
| J-OSLER修了要件 | 症例数（120/160）、疾患群数（56）、病歴要約数（29） | naika.or.jp/nintei/j-osler/ |
| 専門医試験 | 受験資格、日程、合格率 | 各学会公式サイト |
| 税制・控除 | iDeCo掛金上限、確定申告ルール、所得税率 | 国税庁nta.go.jp、厚労省 |
| 育休・産休制度 | 給付金支給率、制度改正 | 厚労省mhlw.go.jp |
| 学会年会費 | 金額、入会要件 | 各学会公式サイト |
| 給与・相場 | 専攻医の年収帯、バイト時給 | 複数ソースで裏取り |

#### ② ファクトチェックの手順

1. 記事内の**すべての数字・金額・制度名・要件**を抽出する
2. 各項目について**公式ソースをWeb検索**で確認する
3. 記事の日付時点で正しい情報に更新する
4. 制度改正の予定がある場合は「〇年〇月施行予定」と明記する
5. 確認できないデータは「目安」「施設により異なる」等のヘッジを入れる

#### ③ やってはいけないこと

- **「〜とされています」「〜と言われています」で具体的な数字を書く** → 必ず出典を確認
- **AIの学習データの金額をそのまま使う** → 制度改正で変わっている可能性大
- **古い制度情報を「2026年最新」として公開する** → 信頼を失う最大のリスク
- **フロントマターのtitle/descriptionに`**`太字マーカーを入れる** → MDXでは効果なし
- **SVGの`<text>`要素内に`**`太字マーカーを入れる** → レンダリングされない

```markdown
## 心不全の治療薬｜HFrEFとHFpEFで異なるアプローチ

心不全は「EF低下型（HFrEF）」と「EF保持型（HFpEF）」の区別が試験の核心です。
HFrEFでは**予後を改善する4種類の薬剤**が確立されており、
試験ではこの4種の使い分けが繰り返し出題されます。

第一の柱は**ACE阻害薬/ARB（またはARNI）**で、
レニン・アンジオテンシン系を抑制して心臓の負荷を軽減します。
第二はβ遮断薬（カルベジロール・ビソプロロール）で、
心拍数を下げて心筋酸素消費量を減らします。
ここで注意すべきは、急性心不全の増悪時にはβ遮断薬は慎重投与ですが、
**安定した慢性HFrEFでは予後改善のために必須**であることです。
この「急性期と慢性期で対応が逆」という点は試験で引っかけ問題として出やすいです。

第三はMRA（スピロノラクトン・エプレレノン）、
第四は近年追加された**SGLT2阻害薬**（ダパグリフロジン・エンパグリフロジン）です。
SGLT2阻害薬は元々糖尿病治療薬ですが、心不全に対する予後改善効果が証明され、
糖尿病の有無にかかわらず使用されるようになりました。
```

> 読者が実際に間違えやすいポイントがある場合に追加する。ツール紹介・スケジュール系など、失敗パターンが不自然な記事では省略してよい。

### 配置場所: まとめセクションの直前

### フォーマット

```markdown
## よくある失敗3パターン

[なぜその失敗が起きるのかを1〜2文で説明]。

...

### ❸ [具体的な失敗の内容]
...
```

### 失敗パターンの探し方

1. **数値の混同**: 「4.5時間」を「6時間」と覚える、「200」を「100」と覚える
2. **治療の逆**: 「ステロイド有効」と「ステロイド無効」の疾患を取り違える
3. **禁忌の見落とし**: 「DLBに抗精神病薬」「褐色細胞腫にβ遮断薬先行」
4. **手順の逆転**: 「血培の前に抗菌薬」「α遮断の前にβ遮断」
5. **分類の混同**: 「CDとUCの内視鏡所見」「GBSとMGの治療」

## STEP 6: 内部リンク（最低3本、目標5本）

### 配置ルール

- **本文中**: 関連テーマに触れた箇所で自然にリンク（2〜3本）
- **まとめ**: 学習計画の文脈で他の領域記事へリンク（2〜3本）

### アンカーテキストのルール

```markdown
❌ NG: [こちら](/blog/xxx)
❌ NG: [この記事](/blog/xxx)
✅ OK: [内科専門医試験の出題傾向と対策](/blog/g04-naika-senmoni-shutsudai-keiko)
✅ OK: [病歴要約の書き方完全ガイド](/blog/c01-josler-byoreki-youyaku-kakikata)
```

### リンク先の選び方

1. **同じクラスター内の記事**（例: H01→H02, H03...）
2. **同じピラー記事**（例: H01→G04出題傾向, G01勉強法）
3. **関連する別クラスターの記事**（例: H09感染症→H08膠原病）

### 内部リンクのURL形式

```markdown
✅ OK（相対パス）: [記事タイトル](/blog/slug)
⚠️ 許容（絶対URL）: [記事タイトル](https://iwor.jp/blog/slug)
```

相対パスを推奨。絶対URLでも問題ないが統一すること。

## STEP 7: コミット前の最終チェックリスト

以下を**すべて**確認してからコミットすること。1つでも未達の場合はコミットしない。

```bash
# 1. SVGが2個以上あるか
count=$(grep -c '<svg' content/blog/NEW_ARTICLE.mdx)
[ "$count" -ge 2 ] && echo "✅ SVG: ${count}個" || echo "❌ SVG不足: ${count}個"

# 2. CTAが2箇所以上あるか
cta=$(grep -c 'booth.pm' content/blog/NEW_ARTICLE.mdx)
[ "$cta" -ge 2 ] && echo "✅ CTA: ${cta}箇所" || echo "❌ CTA不足: ${cta}箇所"

# 3. 内部リンクが3本以上あるか
links_rel=$(grep -oE '\(/blog/[^)]+\)' content/blog/NEW_ARTICLE.mdx | wc -l)
links_abs=$(grep -oE 'iwor\.jp/blog/' content/blog/NEW_ARTICLE.mdx | wc -l)
links=$((links_rel + links_abs))
[ "$links" -ge 3 ] && echo "✅ 内部リンク: ${links}本" || echo "❌ リンク不足: ${links}本"

# 4. よくある失敗セクションがあるか（推奨・省略可）
fail=$(grep -c 'よくある失敗' content/blog/NEW_ARTICLE.mdx)
[ "$fail" -ge 1 ] && echo "✅ 失敗パターンあり" || echo "⚠️ 失敗パターンなし（文脈上有益なら追加推奨）"

# 5. 5行以上の箇条書き壁がないか（リファレンスリスト除く）
walls=$(awk '/^- |^[0-9]+\. |^\*\*[A-Z]/{c++; next} c>=5{n++} {c=0} END{print n+0}' content/blog/NEW_ARTICLE.mdx)
[ "$walls" -eq 0 ] && echo "✅ 箇条書き壁なし" || echo "⚠️ 箇条書き壁${walls}箇所（リファレンスリストなら許容）"

# 6. 文字数が十分か（SVG含む）
chars=$(wc -c < content/blog/NEW_ARTICLE.mdx)
[ "$chars" -ge 12000 ] && echo "✅ 文字数: ${chars}" || echo "❌ 文字数不足: ${chars}"
```

### チェックリスト（目視確認）

```
□ 各H2セクション冒頭に2〜3段落の散文があるか？
□ 散文に「なぜ」「試験でどう問われる」「何を間違えやすい」のいずれかが含まれるか？
□ 太字が文中に自然に配置されているか（1段落1〜2箇所）？
□ SVGが2種類以上あり、テンプレの使い回しでないか？
□ CTAが冒頭blockquote + 中盤 + まとめ付近の最低2箇所あるか？
□ 「よくある失敗3パターン」セクションは文脈上有益か（有益なら配置、不要なら省略可）？
□ 内部リンクが本文中2本 + まとめ2本 = 最低3本以上あるか？
□ ファクトチェック（CONTENT_GUIDE §11.19）を実施したか？
```

### After（修正後の高品質パターン）

```markdown
## 虚血性心疾患の要点

### STEMI vs NSTEMIの鑑別ポイント

虚血性心疾患の急性期治療で試験に最も出るのは、STEMIとNSTEMI
の鑑別です。鑑別の最も重要な基準は**心電図でのST上昇の有無**です。

**STEMI（ST上昇型心筋梗塞）**は、12誘導心電図で**ST上昇が
隣接する2誘導以上**に認められる場合に診断されます。責任病変は
完全閉塞であることが多く、治療の第一選択は**緊急PCI
（door-to-balloon time 90分以内）**です。「door-to-balloon
90分以内」という数値は試験で頻出なので暗記してください。

**NSTEMI（非ST上昇型心筋梗塞）**は、ST低下・陰性T波、または
ST変化がない状態で**トロポニンが上昇**しているケースです。
トロポニン上昇の有無が不安定狭心症との鑑別ポイントになります。
NSTEMIではリスク層別化を行ったうえでPCIの適応を判断するため、
STEMIほどの緊急性はありませんが、高リスク例には24時間以内の
早期侵襲的戦略が推奨されます。
```

**改善点:**
- 散文で「なぜこの鑑別が重要か」を説明
- 「試験で頻出」「暗記してください」など読者への具体的な助言
- 太字が文中の重要キーワードに自然に配置
- 箇条書きではなく散文で情報を展開

## 追加ルール: SVG要素の重なり防止（2026年3月13日追加）

### 問題

モバイル表示でSVG内のテキスト・円・線（接続線）が重なって読めなくなる。特に:
- 図形の辺の上にテキストを配置すると、線とテキストが重なる
- 著作権フッターが円の下端と重なる
- viewBoxの高さが不足して要素が詰まる

### ルール

1. **テキストには背景カード（rect）を敷く**: 線や図形の上にテキストを置く場合、テキストの後ろに白背景の`rect`を配置して可読性を確保する
2. **viewBoxの高さに余裕を持たせる**: 最下部の要素（円の下端 or テキスト）から著作権フッターまで**最低40px**の余白を確保する
3. **円の半径とテキスト位置の関係**: `circle cy + r`（円の下端）と次の要素の`y`座標の間に**最低20px**の余白を確保する
4. **接続線はテキストを避ける**: `line`要素の始点・終点がテキスト要素と重ならないよう、座標を調整する
5. **モバイルでの確認**: SVGの幅600〜800pxをモバイル画面幅（375px）に縮小した時のレイアウトを想像して設計する

### チェック方法

```
viewBoxの高さ ≥ 最下部要素の(y + r or font-size) + 40px + フッターテキストの高さ(20px)
```

## 追加ルール: 見出しにカテゴリプレフィックスを入れない（2026年3月13日追加）

### 問題

`## K01: バーンアウト対策` のように、H2見出しに記事IDやクラスター名のプレフィックスが入ると、読者には意味不明な「K01」が表示される。

### ルール

- **H2/H3見出しにファイルID（A01, B02, K07等）やクラスター記号を含めない**
- 見出しは読者が読む自然な日本語にする
- SVG内のテキストにもID/クラスター記号を含めない
- 内部リンクのアンカーテキストにもID/クラスター記号を含めない

```markdown
❌ NG: ## K01: バーンアウト対策
❌ NG: [K01: バーンアウト対策](/blog/k01-burnout-taisaku)
✅ OK: ## バーンアウト対策
✅ OK: [バーンアウト対策](/blog/k01-burnout-taisaku)
```

## 追加ルール: 太字マークダウンの括弧ルール（2026年3月13日追加）

### 問題

remarkパーサーの仕様で、`**`の内側に全角括弧`（）`があると太字として認識されず、`**`がそのまま文字として表示される。

### ルール

```markdown
❌ NG: **テキスト（説明）**        → **がそのまま表示される
❌ NG: **テキスト（説明）**の後に   → 同上
✅ OK: **テキスト**（説明）        → 括弧を**の外に出す
✅ OK: **テキスト** （説明）       → スペースを入れてもOK
```

**太字の中に全角括弧`（）`を入れてはいけない。括弧は必ず`**`の外に出す。**

### チェックコマンド（コミット前に実行）

```bash
grep -rn '\*\*[^*]*（[^*]*）\*\*' content/blog/NEW_ARTICLE.mdx
# 出力があれば修正が必要
```

## 追加ルール: テーブルのモバイル対応（2026年3月13日追加）

### 問題

MDXテーブルのカラム数が5以上あると、モバイル（375px幅）で横にはみ出して読めなくなる。

### ルール

- テーブルのカラム数は**4以下を推奨**
- 5カラム以上が必要な場合は、テーブルを分割するか、散文で情報を伝える
- CSS側で`overflow-x: auto`を設定済みだが、横スクロールは読者体験が悪いため避けるべき
