## 1. コンセプト

**「QB＋Anki＋AI家庭教師」の統合体を、基本無料で提供する**

- 全過去問演習: 無料（ユーザー獲得ファネル）
- AI解説・カード生成・苦手分析: PRO（クレジット課金）
- medu4終了＋国試CBT化のタイミングで市場の空白を取る

## 2. 法務リスク評価（確定）

| 要素 | リスク | 根拠 |
|---|---|---|
| 問題文＋選択肢＋正答の公開 | 極めて低い | 厚労省が自らPDF公開。medu4/QB/MEC等が商用利用、厚労省から権利主張事例ゼロ |
| AI生成の解説 | ゼロ | iwor studyのオリジナル著作物 |
| 公開画像の利用 | 低い | 厚労省公開分のみ使用 |
| 科目分類・タグ付け（自前） | ゼロ | 独自体系 |
| 出典表記 | 必須 | 「厚生労働省 第XXX回医師国家試験より」を全問に表記 |

**medu4からのデータ取得方針:**
- 解説文: 取得しない
- 正答率: 取得しない（自前で蓄積）
- 科目・テーマ・形式: 検索結果一覧を手動コピペ（著作権リスク極小）
- 問題文: medu4からではなく厚労省PDFから取得

**mediLink利用規約 第7条(3):** 複製・二次著作物作成を明示的に禁止。QBのスクショ取り込み等の機能は作らない。

## 3. データ構築パイプライン（確定）

### 3.1 ソースとフロー

```
[厚労省PDF] ──テキスト抽出（deterministic）──→ 問題文・選択肢
[厚労省PDF] ──別PDF──→ 正答
[medu4検索結果] ──手動コピペ──→ 科目・テーマ・形式・正答率
                                    ↓
                              統合JSON/DB
                                    ↓
                         Claude APIで分類タグ付与（iwor独自体系）
                                    ↓
                         関連問題リンクをバッチ生成
```

### 3.2 厚労省PDFの特性

- デジタルテキスト埋め込みPDF（OCR不要）
- 問題本文のテキスト抽出: pdfplumber / pymupdf で可能（実証済み: 119A全75問を抽出成功）
- 別冊（画像問題）: 画像として抽出。一部は厚労省が非公表/加工済み
- 正答: 別PDFで公開。テーブル形式、パース容易
- 対象回数: 100回〜120回（medu4の収録範囲と一致）

### 3.3 ハルシネーション対策

- Phase 0（テキスト抽出）: AIを使わない → ハルシネーションゼロ
- Phase 0.5（分類タグ付与）: AIだが、分類の誤りは致命的でない（後から修正可能）
- Phase 1（解説生成）: 正答は厚労省データ（AIが判断しない）。解説のみAI生成
- 3層防御: ①正答は公式データ ②静的解説は医師レビュー ③対話はRAG＋免責表示

## 4. 分類体系（仮説 — Phase 0.5でボトムアップ再設計）

### 4.1 臨床領域（Clinical Domain）— 20分類

### 4.2 タグ体系（仮説 — Phase 0.5で全問データから帰納的に導出）

- 臨床アクションタグ: 暫定6種（診断/検査選択/治療選択/病態理解/予防/対応・説明）→ 全問を見て再設計
- 知識タイプタグ: 暫定5種（事実/機序/臨床判断/計算/英語）→ 全問を見て再設計
- カードタイプ: 暫定5種（事実/機序/鑑別/臨床判断/ゴロ）→ 全問を見て再設計

**重要: これらは直感ベースの仮説。Phase 0.5で8,000問をClaude APIに「自由記述で」分類させ、クラスタリングして実データに基づく体系を構築する。**

## 5. AI解説生成（確定）

### 5.1 モデル使い分け

| 処理 | モデル | コスト |
|---|---|---|
| 静的解説（事前生成、1回） | Opus 4 | ~24,000円/全8,000問 |
| 対話型解説 | Sonnet 4 | クレジット消費 |
| カード生成 | Haiku 4.5 | クレジット消費 |
| 苦手分析レポート | Sonnet 4 | クレジット消費 |
| 関連問題リンク | バッチ処理済み | API不要 |

### 5.2 プロンプト設計の要点

- 正答はAIが判断しない（厚労省データを注入）
- 各選択肢に○/×＋理由を付与
- 背景知識はイヤーノートレベルの体系的まとめ
- 対話時はユーザーの演習データ（苦手領域・正答率）をsystem promptに注入
- カード生成は解説から「覚えるべき知識」を抽出→複数カードに分解

→ プロンプト全文は iwor_study_design.md（ローカル設計書）参照

## 11. ライフサイクル・ロックイン戦略

### 11.1 コアコンセプト: 「試験ツール」ではなく「生涯知識基盤」

iwor studyは試験対策アプリではない。**医師のキャリア全体を通じて知識を蓄積・活用するプラットフォーム**。

解約したら全データが消える。蓄積された学習データ・カード・ノート・論文ライブラリが全て消えるため、解約の心理的コストが極めて高い。

### 11.2 ステージ別の提供価値

```
医学部3-4年（CBT）
  └─ CBT過去問演習 + カード生成 + 苦手分析
       ↓ データ全引き継ぎ
医学部5-6年（国試）
  └─ 国試過去問演習 + AI解説 + 臨床推論トレーニング
  └─ CBT時代のカードが「国試の基礎」として復習に使える
       ↓ 国試合格直後に「初期研修モード」への自動切替提案
初期研修
  └─ 当直サバイバルカード（国試知識から臨床知識に自動変換）
  └─ 臨床ツール（計算機、薬剤ガイド）→ 使うたびにStudyへの導線
  └─ 研修記録（EPOC）がStudyと同じプラットフォーム内
  └─ 個人ナレッジベース（Notion的メモ・PDF保存）に臨床メモを蓄積
       ↓ 専門科決定時に「専攻医モード」への切替提案
専攻医
  └─ 専門医試験過去問QB（分野横展開）
  └─ 論文管理（Zotero/Endnote代替）← 現在の論文フィードを進化
  └─ J-OSLER管理がStudyと同じプラットフォーム内
  └─ 過去の全学習データが参照可能（「学生時代に学んだ循環器の知識」）
       ↓
一般医師/指導医
  └─ 年次ガイドライン更新カード（AIが差分を抽出→カード化）
  └─ 論文管理は継続利用（Zotero移行コスト高い）
  └─ 後輩の指導時に「この疾患のカード」を共有
```

### 11.3 ロックイン要素

| 要素 | ロックイン強度 | 説明 |
|---|---|---|
| 学習データ蓄積 | ◎ | 正答率推移、苦手分析、AI対話履歴。移行不可能 |
| カードライブラリ | ◎ | 数千枚のカード＋忘却曲線パラメータ。エクスポートしてもFSRSデータは移行不可 |
| 個人ナレッジベース | ◎ | Notion的な構造化メモ＋PDF保存。移行コスト極大 |
| 論文ライブラリ | ◎ | 論文管理＋タグ＋メモ。Zoteroへの移行は可能だが面倒 |
| 臨床ツールのお気に入り | ○ | PRO: よく使うツールをお気に入り→即呼び出し。HOKUTOの代わりにiworを開く習慣 |
| 研修記録（EPOC/J-OSLER） | ○ | 制度上必須のデータがiwor上にある |
| 同期ネットワーク | ○ | 同期の学習進捗が見える。ソーシャル離脱コスト |

### 11.4 論文フィード → 論文管理アプリへの進化

現在の `/journal/` は論文フィード（RSS的な閲覧のみ）。これを**Zotero/Endnote代替**に進化させる。

```
現在: 論文フィード（閲覧のみ）
  ↓
Phase 1: 論文保存＋タグ付け（お気に入りに保存、フォルダ/タグ管理）
Phase 2: PDF保存＋注釈（サーバーにPDF保存、ハイライト、メモ）
Phase 3: 引用生成（Vancouver/APA/Harvard形式で引用文を自動生成）
Phase 4: AIサマリー＋カード化（論文からkey findingsをカード生成）
Phase 5: 共有ライブラリ（研究室・科内で論文を共有）
```

**既存のZotero/Endnoteにない差別化:**
- 論文の知見をStudyカードに変換できる（「この論文のkey findingを覚えたい」→カード化）
- 臨床ツールと連携（「eGFRの計算に使ったガイドラインの原著論文」が紐付く）
- AIが論文をサマリーし、「あなたの専門分野の最新論文5本」を週次レポート

### 11.5 個人ナレッジベース（Notion的機能）

```
ユーザーがiworに保存できるもの:
├─ テキストメモ（Markdown対応）
├─ PDF（講義スライド、教科書ページ、論文）
├─ 画像（臨床写真、病理画像、心電図）
├─ フラッシュカード（既存FSRS機能）
└─ 過去問の演習メモ（「この問題のポイント」）

すべてが相互リンク可能:
- メモ → 関連カード
- 論文 → 関連メモ → 関連過去問

```

**Notionとの違い:** 汎用ツールではなく**医学に特化**。疾患名・薬剤名・検査名をタグとして自動認識し、関連コンテンツを自動リンク。

## 13. データベース・バックアップ戦略

### 13.1 D1の制約

- 1データベースあたり10GB上限（課金しても増やせない）
- 1アカウントで最大50,000データベース作成可能
- シングルスレッド（1クエリずつ逐次処理）
- Time Travel: 30日間のポイントインタイムリカバリ（自動）

### 13.2 DB分割設計

用途別にD1を分ける。初期は1つで開始し、10GBに近づいたら分割。

```
Phase 1（初期〜1年目）: d1-study 1つで全テーブル（~2GB）
Phase 2（1年目〜）: 演習記録が肥大化したら分割
  d1-core:     問題マスタ、解説、分類タグ（~200MB、読み取り中心）
  d1-users:    ユーザー情報、クレジット、設定（~100MB）
  d1-attempts: 演習記録（最大。年別パーティション可能）
  d1-cards:    カード＋FSRSパラメータ（~3GB）
  d1-notes:    メモ・論文メタデータ（~500MB）
```

### 13.3 バックアップ

| 層 | 方法 | 保持期間 |
|---|---|---|
| 第1層 | D1 Time Travel（自動） | 30日 |
| 第2層 | 日次SQLダンプ→R2保存（Scheduled Worker） | 90日 |
| 第3層 | 問題マスタ＋解説のJSON→R2（変更時のみ） | 永続 |
| 第4層 | クレジット残高のKV二重書き | リアルタイム |
