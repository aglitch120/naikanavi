> 最終更新: 2026年3月9日
> 目的: 日本の読者に最も有益で読まれるブログ記事を一貫した品質で制作する

## 1. コンテンツ制作の基本原則

### 1.1 E-E-A-T を全記事で体現する

| 要素 | 内科ナビでの実装 |
|------|----------------|
| **Experience（経験）** | J-OSLER 8期生としての実体験、修了済みの知見を記事に反映 |
| **Expertise（専門性）** | 医学的正確性、J-OSLER公式情報との整合性を必ず担保 |
| **Authoritativeness（権威性）** | 著者プロフィール、参考文献・引用、公式PDFへのリンク |
| **Trustworthiness（信頼性）** | 情報の出典明示、更新日表示、誤情報の訂正体制 |

### 1.2 SEO + GEO（生成AI最適化）二刀流

2026年のコンテンツ戦略は、Google検索と生成AI検索の両方に対応する必要がある。

**SEO対策（従来型検索）**
- キーワードをタイトル・H2・冒頭100字に自然配置
- 内部リンク構造（ピラー⇔クラスター）
- 構造化データ（Article, FAQPage, BreadcrumbList）
- Core Web Vitals の最適化

**GEO対策（AI検索最適化）**
- 冒頭で「〜とは」形式の明確な定義文を置く（AIが引用しやすい）
- 数値・統計データを具体的に記載（「29症例」「56疾患群」等）
- FAQ セクションを必ず設置（AIが構造化情報を好む）
- 比較表・手順リストで情報を構造化
- 一次情報（体験談・独自データ）を積極的に含める

### 1.3 記事制作フロー

```
1. キーワード確認 → docs/keywords/naikanavi_keyword_list.xlsx 参照
2. 【必須】Web検索で最新情報を収集（下記「執筆前リサーチ」参照）
3. アウトライン作成（H2/H3構造）
4. 本文執筆（本ガイドライン準拠）
5. 【必須】ファクトチェック（下記「公開前ファクトチェック」参照）
6. サムネイル・図解SVG作成
7. MDファイルをチャット内に提示 → ユーザーのOK取得
8. リポジトリに push
```

### 1.4 【必須】執筆前リサーチ（Web検索）

**記事を1文字でも書く前に、必ず以下のWeb検索を実施する。**

#### a) 検索意図の確認
- 対象キーワードでGoogle検索し、上位3〜5記事の構成・見出し・文字数を確認
- 「読者は何を知りたくてこのキーワードで検索しているか」を把握
- 競合が書いていない切り口・不足情報を特定（差別化ポイント）

#### b) 最新情報の確認
- J-OSLER公式サイト（naika.or.jp）の最新PDF・FAQ・お知らせ
- 内科学会・専門医機構の制度変更・スケジュール変更
- 該当年度の症例数・要件変更（120症例 vs 160症例 等）
- 試験日程・合格率など最新年度のデータ

#### c) ユーザーの悩み収集
- `J-OSLER [キーワード] site:twitter.com` or `site:x.com`
- Yahoo!知恵袋・医師向けQ&Aでのリアルな質問
- 競合ブログのコメント欄

#### d) 競合の強み・弱みの分析
- おちば先生（ikuji-doctor.com）: テンプレ配布・網羅的
- ゼロえん先生（zero-doctor.com）: 体験談ベース
- かめちゃん（doctor-turtle.com）: Excelチェックリスト配布
- **内科ナビの差別化**: AIツール連携、進捗ダッシュボード、全方位カバー

**検索例:**
```
J-OSLER 病歴要約 2026 変更点
J-OSLER 専攻医 悩み site:twitter.com
内科専門医 病歴要約 差し戻し
[対象キーワード] site:ikuji-doctor.com
[対象キーワード] site:naika.or.jp
```

### 1.5 【必須】公開前ファクトチェック

**記事完成後、公開前に以下を必ず検証する。**

#### a) 医学的正確性
- 疾患名・薬剤名・検査名のスペル・正式名称を確認
- 診断基準・治療指針が最新のガイドラインと一致しているか
- 引用した数値データ（合格率・症例数等）の出典を確認

#### b) J-OSLER制度情報の正確性
- 症例数（120 or 160）、病歴要約数（29）が正しいか
- 期限・締切日が最新年度のものか
- 評価フロー（個別→一次→二次）の記述に誤りがないか
- J-OSLER公式PDF・FAQと矛盾していないか

#### c) リンク・参照の検証
- 内部リンクのURLが正しく、404にならないか
- 外部リンク（公式サイト等）が有効か
- CTAリンク（BOOTH）が正しいか

#### d) 競合との差別化確認
- 競合記事の単なるリライトになっていないか
- 内科ナビならではの独自価値（体験談・AI連携・ツール訴求）が含まれているか

## 2. 日本のブログ読者に読まれる文章の書き方

### 2.1 冒頭3秒で心をつかむ

日本のWeb読者の離脱は冒頭で決まる。以下の構造で書き出す。

```
① 共感フック（読者の心の声を「」で表現）
② 解決の約束（この記事を読むとどうなるか）
③ 著者の信頼性（なぜこの人の話を聞くべきか）
```

**例:**
```markdown
「病歴要約、3回連続で差し戻された...もうどこを直せばいいかわからない」

こんな経験はありませんか？

この記事では、J-OSLER 8期生として29症例を効率的に完了した筆者が、
Accept率を劇的に上げる具体的なテクニックを解説します。
```

### 2.2 文章のリズムとリーダビリティ

| ルール | 基準 |
|--------|------|
| 一文の長さ | **60文字以内**（句読点まで） |
| 改行 | **2〜3文ごと**に空行 |
| 漢字率 | **30〜40%**（漢字が多すぎると圧迫感） |
| 段落の長さ | **最大5行**（スマホで画面いっぱいにならない長さ） |
| 箇条書き | **1箇所5個以内**（多すぎると読み飛ばされる） |

### 2.3 PREP法ベースの段落構成

各H2セクションは以下の構造で書く。

```
P（Point）: 結論を最初に述べる
R（Reason）: なぜそうなのか理由を説明
E（Example）: 具体例・データ・体験談で裏付け
P（Point）: 結論を再確認し、次のセクションへ繋ぐ
```

### 2.4 読者を退屈させない工夫

- **太字強調**: 各段落で最も重要な1フレーズのみ。乱用しない
- **吹き出し/引用**: 体験談は `>` 引用ブロックで視覚的に区別
- **数字を使う**: 「多くの」→「約7割の」「3つのポイント」
- **比較**: Before/After や NG例/OK例 の対比で理解促進
- **小見出し（H3）**: 長いセクションは300〜500字ごとにH3で分割

### 2.5 語尾のバリエーション

同じ語尾の連続は単調になる。以下を意識的に混ぜる。

```
〜です。 / 〜ます。         ← 基本
〜でしょう。 / 〜かもしれません。 ← 推量
〜ましょう。 / 〜してください。  ← 行動促進
〜ですよね。 / 〜ではないでしょうか。 ← 共感
体言止め。                    ← リズム変化
```

## 3. 記事構成テンプレート

### 3.1 フロントマター（MDX）

```yaml
---
title: "【〇〇】タイトル（40文字以内）"
description: "説明文（120文字以内）"
date: "2026-03-10"
updated: "2026-03-10"
author: "内科ナビ編集部"
category: "medical-history"
cluster: "B"
pillar: "josler-complete-guide"
tags: ["J-OSLER", "病歴要約", "書き方"]
cta_type: "template"
reading_time: 8
status: "published"
seo_checked: true
---
```

### 3.2 記事本文の構成

```
1. 共感フック + リード文（100字以内で価値提案）
2. ソフトCTA（冒頭テキストリンク）
3. 本文 H2×5〜8セクション
   ├── 各 H2 に関連キーワード・共起語を含む
   ├── 各セクション 300〜500字
   ├── H3 で必要に応じて分割
   ├── 文脈内CTA を2箇所以上に自然に配置
   └── 図解SVGを適切な箇所に配置（後述）
4. FAQ セクション（3〜5問、GEO対策）
5. まとめ（3ポイント整理）
6. ハードCTA（末尾）
7. 著者プロフィール
```

### 3.3 タイトルの型（40文字以内）

```
型1: 【カテゴリ】+ KW + 数字 + ベネフィット
例: 【完全ガイド】J-OSLER病歴要約の書き方｜Accept率を3倍にする方法

型2: 【年度】+ KW + 完全/徹底 + 解説
例: 【2026年最新】J-OSLER修了要件を徹底解説｜120症例の進め方

型3: 問いかけ型 + 解決
例: J-OSLER病歴要約が差し戻される？原因と対策を8期生が解説
```

### 3.4 メタディスクリプション（120文字以内）

- 「誰の」「何の悩みを」「どう解決するか」を含める
- 数字を入れる（「29症例」「5つのコツ」等）
- 行動を促す末尾（「〜を解説します」「〜がわかります」）

### 3.5 FAQセクション（GEO対策用）

記事末尾に必ず設置。構造化データとしても出力される。

```markdown
## よくある質問

### Q. J-OSLERの病歴要約は何症例必要ですか？

A. J-OSLERの修了には29症例の病歴要約が必要です。
7期生以降は症例登録数が160→120症例に変更されましたが、
病歴要約の必要数は29症例で変わりません。

### Q. 病歴要約の総合考察では何を書けばよいですか？

A. 総合考察では、①診断に至った臨床推論、②治療方針の根拠、
③全人的視点（患者の社会的背景・QOL）の3点を含めることが重要です。
```

## 4. ビジュアルデザイン（図解・サムネイル）

### 4.1 基本方針

- **文字だけの壁を作らない**: 1スクロール（約600px相当）に1つはビジュアル要素を入れる
- **過剰にしない**: 図解は記事内に2〜4点が目安（サムネ含む）
- **ブランドイメージ統一**: デザインシステム（DESIGN_SYSTEM.md）のカラーを使用
- **抽象的でおしゃれ**: 幾何学図形、フロー図、比較チャートなど。リアルな人物イラストは不要

### 4.2 サムネイル（アイキャッチ）— 必須

全記事に必ずサムネイルを作成する。

**仕様:**
- サイズ: 1280×670px（SVG → rsvg-convert でPNG化）
- クラスター別カラーパレット使用（DESIGN_SYSTEM.md参照）
- フォント: Noto Sans CJK JP, weight 900
- 年度表記は入れない（古びるため）
- ファイル名: `[slug]-thumbnail.png`
- public/blog/images/ に配置

**レイアウト:**
```
┌──────────────────────────────────┐
│  背景色（クラスター別）           │
│                                  │
│   [タグ]（小さめ・角丸背景）      │
│                                  │
│   メインタイトル                  │
│   （80px, 太字, 最大12文字）      │
│                                  │
│   サブタイトル                    │
│   （48px, 半透明）               │
│                                  │
│                    ○ 装飾円      │
│   内科ナビ（24px, 薄め）         │
└──────────────────────────────────┘
```

### 4.3 本文内の図解SVG

記事の理解を助けるための図解を2〜3点作成する。

**図解の種類と使いどころ:**

| 種類 | 使いどころ | 例 |
|------|-----------|-----|
| フロー図 | プロセス・手順 | 病歴要約の提出フロー、症例登録の流れ |
| 比較表 | 2つ以上の選択肢 | Accept vs Reject の違い、旧制度vs新制度 |
| チェックリスト | やるべきこと | 提出前チェック項目 |
| 構造図 | 階層・関係性 | 疾患群の分類、J-OSLER全体像 |
| タイムライン | 時系列 | 専攻医のスケジュール |

**デザインルール:**
- 幅: 記事幅に合わせて800px固定
- 背景: 白 or `#F5F4F0`（ページ背景色）
- アクセント: クラスターカラー or `#1B4F3A`（ブランドカラー）

- 角丸: 8〜12px
- パディング: 十分な余白を確保

- 抽象的・幾何学的なスタイル（ポップすぎない、プロフェッショナルな印象）

## 5. SEOバックエンド対策

### 5.1 各記事で確認するチェックリスト

**フロントマター:**
- [ ] title: 40文字以内、主要KWが前方
- [ ] description: 120文字以内、「誰の」「何の」「どう解決」
- [ ] category/cluster: 正しいカテゴリに配置
- [ ] tags: 関連タグ3〜5個
- [ ] pillar: 所属するピラー記事を指定

**構造化データ（自動生成されるが確認）:**
- [ ] Article スキーマ（title, author, datePublished, dateModified）
- [ ] BreadcrumbList
- [ ] FAQPage（FAQセクションがある場合）

**メタタグ:**
- [ ] canonical URL が正しい
- [ ] robots: max-snippet:-1, max-image-preview:large
- [ ] OGP画像が正しく生成される

### 5.2 内部リンク戦略（毎記事必須）

**1記事あたり3〜5本のリンクを設置:**

```
① ピラー記事 → クラスター記事（親→子）: 1本
② クラスター記事 → ピラー記事（子→親）: 1本  
③ 同クラスター内の関連記事: 1〜2本
④ 異クラスターの関連記事: 0〜1本
```

**アンカーテキストのルール:**
- KWを含む自然な日本語にする
- ✅ 良い例: 「[病歴要約の書き方](/blog/josler-byoreki-kakikata)については別記事で詳しく解説しています」
- ❌ 悪い例: 「[こちら](/blog/josler-byoreki-kakikata)をご覧ください」

**内部リンクの確認:**
- 新記事を追加したら、関連する既存記事にも逆リンクを追加する
- 孤立ページ（どこからもリンクされていない記事）を作らない

### 5.3 ピラー⇔クラスター構造

```
ピラー: J-OSLER完全攻略ガイド (/blog/josler-complete-guide)
├── A: J-OSLER基礎（A01〜A08）
├── B: 症例登録（B01〜B08）
├── C: 病歴要約（C01〜C12）
├── D: 疾患別病歴要約（D01〜D15）
├── E: 進捗管理（E01〜E06）
└── F: JMECC・講習（F01〜F05）

ピラー: 内科専門医試験合格マニュアル (/blog/exam-preparation-guide)
├── G: 内科専門医試験（G01〜G11）
├── H: 試験領域別（H01〜H10）
└── I: 総合内科専門医（I01〜I08）
```

新記事公開時は、該当ピラー記事の目次・リンク集にも追記すること。

### 5.4 URL設計

- 形式: `/blog/[slug]`
- slug: 英数字・ハイフン、KWのローマ字
- 例: `josler-byoreki-youyaku-kakikata`（病歴要約の書き方）
- 不変: 一度公開したURLは変更しない（リダイレクトが必要になるため）

## 6. CTA配置ルール

### 6.1 全体の配置構造

```
記事上部:  ソフトCTA（テキストリンク）
記事中盤:  文脈内CTA × 2箇所（本文に自然に織り込む）
記事下部:  ハードCTA（大型バナー）
```

### 6.2 CTA URL

すべてのCTAリンク先: `https://naikanavi.booth.pm/items/8058590`

### 6.3 文脈内CTAの5パターン

記事の内容に応じて最適なパターンを選択:

| パターン | 使用記事 |
|---------|---------|
| 病歴要約系: AIテンプレート機能アピール | B,C,D クラスター |
| 進捗管理系: ダッシュボード機能アピール | E クラスター |
| 試験対策系: 問題演習機能アピール | G,H,I クラスター |
| 修了要件系: チェックリスト機能アピール | A,F クラスター |
| 汎用: 全機能まとめてアピール | その他すべて |

### 6.4 価値提供と宣伝のバランス

- **価値提供 70%以上 / 宣伝 30%以下**
- CTAは記事の流れの中で自然に登場させる
- 「売り込み感」を出さない。あくまで「読者の課題解決の一手段」として提示

### 6.5 末尾ハードCTAはMDXに埋め込まない（重要）

末尾の大型CTA（ハードCTA）は `app/blog/[slug]/page.tsx` の `<CTABanner cta={cta} variant="large" />` で**自動挿入される**。MDX本文の末尾にHTMLやJSXでCTAを手動で書くと、自動挿入分と**二重表示**になる。

**ルール:**
- MDX本文に末尾ハードCTAのdiv/JSXを埋め込んではならない
- ソフトCTA（冒頭テキストリンク `>` 引用ブロック）と文脈内CTA（本文中のテキストリンク）のみMDXに記載する
- ハードCTAの内容・デザインを変更したい場合は `lib/blog-config.ts` の `ctaConfig` を編集する

## 11. 病歴要約・症例登録の例文・テンプレート記載ルール

ブログ記事内で病歴要約または症例登録の例文・テンプレート・サンプル文を掲載する場合は、**必ず [`docs/JOSLER_BYOREKI_GUIDELINE.md`](./JOSLER_BYOREKI_GUIDELINE.md) を参照**し、そのルールに従うこと。

ブログ本文（解説パート）は通常のですます調で問題ないが、**J-OSLERに提出する文書の例文として示す部分**は実際のjosler提出時の文体に合わせる必要がある。

**主な注意点（詳細はJOSLER_BYOREKI_GUIDELINE.mdを参照）:**
- 「認める」「にて」禁止 → 「確認する」「で」等に置き換え
- 句読点は「,」「.」を使用（「、」「。」は使わない）
- 体言止め禁止 → 述語で文を閉じる
- 常体・過去形・受動態で記述
- 薬剤名は一般名
- 数字と単位の間に半角スペース

**適用範囲:**
- ✅ コードブロック内の病歴要約テンプレート・例文, NG例/OK例, 疾患別サンプル文
- ✅ 症例登録の「症例の概略」テンプレート・例文
- ✅ 症例登録の「自己省察」の例文（提出文書として示す場合）
- ❌ ブログ解説本文, J-OSLER制度説明, FAQ回答文
- ❌ 自己省察のテンプレート（穴埋め形式で具体的文章がない場合）
