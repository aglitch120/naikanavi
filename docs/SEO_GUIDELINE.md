## 2. 記事作成ルール

### 2.1 タイトル（titleタグ）
- **40文字以内**
- **主要キーワードを前方に配置**
- 【】を使う場合は冒頭に
- 年度を入れる場合は「2026年最新」「2026年版」

```
✅ 良い例
【2026年最新】J-OSLER病歴要約の書き方完全ガイド｜Accept率を上げるコツ

❌ 悪い例
病歴要約の書き方について解説します｜J-OSLER対策
```

### 2.2 メタディスクリプション
- **120文字以内**
- 「誰の」「何の悩みを」「どう解決するか」を含める
- CTAを含める（〜を解説、〜がわかる）

```
✅ 良い例
J-OSLER病歴要約の書き方を査読委員経験者が徹底解説。総合考察・全人的視点の書き方から、差し戻しを防ぐコツまで。テンプレート付きで今日から使えます。
```

### 2.3 見出し構成
```
H1: タイトル（1つのみ）
├── H2: 大見出し（5〜8個）
│   ├── H3: 中見出し
│   │   └── H4: 小見出し（必要な場合のみ）
```

- H2に関連キーワード・共起語を含める
- H2の最初に「結論」「まとめ」を持ってこない（最後に配置）

### 2.4 本文の書き方

#### 冒頭（リード文）
- **100字以内で「誰の何の悩みを解決するか」を明示**
- 読者の共感を得るフック
- 記事を読むメリット

```
✅ 良い例
「J-OSLERの病歴要約、何度書いても差し戻される...」そんな悩みを抱えていませんか？
この記事では、査読委員を8年務めた私が、Acceptされる病歴要約の書き方を徹底解説します。
```

#### 本文構成
- 一文は60文字以内を目安
- 3〜4文ごとに改行
- 箇条書きは5個以内
- 具体例・データを必ず入れる

#### 文字数目安
| 記事タイプ | 文字数 |
|-----------|--------|
| 通常記事 | 3,000〜5,000字 |
| ピラー記事 | 5,000〜8,000字 |
| ツール紹介 | 2,000〜3,000字 |
| Q&A形式 | 1,500〜2,500字 |

### 2.5 画像・図解
- アイキャッチ画像必須（1200×630px推奨）
- 本文中に2〜3枚の図解
- ALT属性に必ずキーワードを含める
- ファイル名は英数字のみ（josler-byoreki-template.png）

### 2.6 内部リンク
- **1記事あたり3〜5本**
- 関連記事への自然な導線
- ピラー記事とクラスター記事を相互リンク
- アンカーテキストにキーワードを含める

### 2.7 CTA配置
| 位置 | タイプ | 目的 |
|------|-------|------|
| 記事上部 | テキストリンク | 即座にツールへ |
| 記事中盤 | バナー | 読み進めた人への提案 |
| 記事下部 | 大型CTA | 記事完読者への訴求 |
| サイドバー | 固定バナー | 常時表示 |

## 11. 記事作成の品質チェック（2026年3月追加）

### 11.1 文字数の最適化
- **ピラー記事**: 5,000〜8,000字（本文のみ、frontmatter除く）
- **通常記事**: 3,000〜5,000字
- **ツール紹介**: 2,000〜3,000字
- 長ければ良いわけではない。**冗長な繰り返しは削除**し、検索意図を満たす最小限の情報量を目指す
- 同じ情報を別の言い回しで繰り返さない

### 11.2 リンク戦略（必須）

#### 内部リンク（3〜5本/記事）
- 関連する既存記事へ**自然な文脈で**リンク
- アンカーテキストにキーワードを含める
- ピラー↔クラスター間の相互リンクを意識

#### 外部リンク（2〜3本/記事）
- **公式ソースを必ずリンク**（日本内科学会、J-OSLER公式ページ等）
- 権威性の高い一次ソースを優先
- `target="_blank" rel="noopener noreferrer"` をつける
- 競合サイトへは原則リンクしない

### 11.3 CTA設計（3箇所必須）

#### 冒頭CTA（inline）
- **ユーザーが「え、マジ？」と思う訴求**にする
- 「テンプレートを無料ダウンロード」ではなく「AIが病歴要約の下書きを30秒で生成」のような具体的ベネフィット
- 数字を入れる（30秒、10分の1、など）

#### 記事中インラインCTA（2〜3箇所）
- 自然な文脈でハイパーリンクとして設置
- 引用ブロック（`>`）で目立たせる箇所は1箇所まで
- 残りは文中に自然にリンクを溶け込ませる

#### 末尾CTA（large/ヒーロー風）
- SVGイラスト付き
- バッジ（ブランド名）
- 3つの特徴・メリットをチェックマーク形式で
- 明確なボタン（「無料で試してみる」等）
- 補足テキスト（「無料トライアルあり。クレジットカード不要。」等）

### 11.4 画像・図解ルール

#### SVG図解（2〜3点/記事）
- DESIGN_SYSTEM.mdのカラーパレットに準拠
- カード内にミニアイコン（クリップボード、ペン、チェックマーク等）でポップさを出す
- `© 内科ナビ naikanavi.com` のフッターを入れる
- ALT属性に具体的な説明を入れる

#### OGP画像
- 新記事追加時は `app/blog/[slug]/opengraph-image.tsx` のtitleMapに追記すること

### 11.5 目次の長さへの配慮
- H2は5〜7個を目安（目次が長くなりすぎない）
- H3は各H2配下に2〜3個まで
- 目次コンポーネントは折りたたみ式（アクティブH2のH3のみ表示）で実装済み

### 11.6 OGP画像（slugToMetaマップ）の更新
- 記事を新規追加したら、必ず `app/blog/[slug]/opengraph-image.tsx` の `slugToMeta` にslug・title・subtitle・categoryを追加する
- OGP画像デザイン: **緑背景（#1B4F3A）**、カテゴリバッジ（白半透明）、大きな白文字タイトル、グリーンサブタイトル、右に装飾円
- Noto Sans JP Boldフォントを動的fetchして太字を正しくレンダリング

### 11.7 アイキャッチ画像
- OGP画像を記事ページ冒頭にアイキャッチとして自動表示
- **配置順序: パンくず → カテゴリバッジ+◯分で読める → アイキャッチ → タイトル+メタ情報 → CTA → 目次 → 本文**
- `rounded-xl overflow-hidden` でデザイン統一

### 11.8 冒頭目次（InlineTableOfContents）
- **モバイルのみ**（`lg:hidden`）で記事冒頭に折りたたみ式目次を表示
- デフォルトではH2を4件までプレビュー表示、「...他N件を表示」で展開
- 展開時はH2+H3すべてを表示、クリックで該当箇所にスクロール後自動で閉じる
- **PCはサイドバー目次（スクロール追従）のみ**

### 11.9 シェアボタンのモバイル対応
- `flex-wrap`で折り返し可能に
- タップターゲット最小44px（`min-h-[44px]` + `px-3 py-2`）
- 「共有:」ラベルはモバイルでは単独行

### 11.10 フッター
- main + footerのコンテナは `px-6 sm:px-8` で余白統一
- フッター下部（ロゴ・リンク・コピーライト）は**左揃え統一**（`items-start`、`text-center`不使用）

### 11.11 CTAのアニメーション
- 末尾CTA（large）: `ctaFloat` アニメーション（0.6s, translateY 12px→0, ease-out）
- 冒頭CTA（inline）: `ctaFloat` アニメーション（0.4s, 同上）
- サブリミナルなレベルの微細な浮き上がり。派手にしない

### 11.12 スマホ冒頭CTAの簡潔さ
- モバイルではdescription（説明文）を非表示（`hidden sm:block`）
- タイトル + ボタン のみのシンプルな構成
- パディング・フォントサイズもPC比で縮小

### 11.13 関連記事カード（ArticleCard compact）
- OGP画像をサムネイルとして表示（`/blog/[slug]/opengraph-image`）
- カード構成: サムネ画像 → カテゴリバッジ → タイトル

### 11.14 マークダウン太字の注意
- `**テキスト（全角括弧）**` → MDXパーサーで壊れる場合がある
- **全角括弧は`**`の外に出す**: `**テキスト**（括弧）`
- 公開前にプレビューで太字が正しくレンダリングされているか確認

### 11.15 タグ
- `/blog/tag/[tag]` でタグページ実装済み（2026-03-10）
- 記事ページのタグは `<Link>` でタグページへ遷移
- ホバー時に `bg-acl text-ac` でインタラクション表示
- `generateStaticParams` で全タグをSSG対応

### 11.16 Edge Runtimeファイルの日本語制限
- `opengraph-image.tsx` 等 `export const runtime = 'edge'` を含むファイルでは、**オブジェクトキーにASCII文字のみ使用**（esbuildが中黒 `・` 等のUnicodeキーをパースできないため）
- 値（value）に日本語を使うのはOK
- カテゴリ等の日本語表示が必要な場合は、ASCIIスラッグ → 日本語のマッピング関数を使う

- 紹介は1記事あたり2〜4箇所が目安。それぞれ異なる文脈（進捗管理 / テンプレート / 演習 etc.）で
- 商品の具体的機能（ダッシュボード、AI病歴要約テンプレート等）に言及し、抽象的な「AIで効率化」のような表現は避ける

### 11.18 内部リンクのバックフィル（逆方向リンク）

新しい記事を作成した際は、**既存記事にも新記事へのリンクを追加する**こと。

#### ルール
- 新記事公開時に、関連する既存記事のまとめセクションや本文中の自然な文脈に、新記事への内部リンクを追加する
- 古い記事ほど内部リンクが少なくなる問題を防ぐ
- 1記事あたり最低1箇所は逆方向リンクを追加
- 無理に入れるのではなく、読者にとって有用な導線であること

### 11.19 ファクトチェック（必須フェーズ・省略厳禁）

記事作成の**最終工程**として、**必ずファクトチェックフェーズを実施する**こと。記事を書き終えた後、コミット・プッシュする**前に**このフェーズを完了させること。AI（Claude含む）の記憶・推測に基づく記載は、公式情報と矛盾するリスクが高い。

#### ⚠️ 作業フロー上の位置づけ
```
記事執筆 → 内部リンク追加 → CTA配置 → OGP更新
→ ★ファクトチェック（ここ！）★
→ コミット → プッシュ → キーワードリスト更新
```
**ファクトチェックを飛ばした場合、誤情報が公開される。必ずこの順序を守ること。**

#### ルール
- **J-OSLER関連の数値・条件・ルール**は、必ず公式ソースをWeb検索・取得して裏取りする
  - [病歴要約 作成の手引き（2023年10月版）](https://www.naika.or.jp/wp-content/uploads/2023/10/2e5a70e7ec35fd38bfba84276602ddd2.pdf)

  - [病歴要約サンプル（2024年7月版）](https://www.naika.or.jp/wp-content/uploads/2024/07/35a75d9e2a04130148491c77a1037343.pdf)
  - [病歴要約29症例の細則](https://www.naika.or.jp/wp-content/uploads/2023/12/byoreki29_saisoku.pdf)
  - [日本内科学会 修了要件ページ](https://www.naika.or.jp/nintei/j-osler/requirements/)
- **「〜とされています」「〜が原則です」等の断定的記述**は、ソースが特定できない場合は書かない
- 記事内のFAQ・数値（症例数、提出上限、条件等）は**1つ1つソースと照合**する
